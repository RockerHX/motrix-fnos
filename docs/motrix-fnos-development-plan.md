# 飞牛版 Motrix 阶段性开发计划

> 目标：在飞牛 OS 上实现一个可安装、可运行、可维护的 GUI 下载工具，核心下载引擎采用 **Aria2 Next**，应用形态参考 Motrix Next：https://github.com/AnInsomniacy/motrix-next，但面向飞牛场景重新设计。

## 1. 当前情况

目前飞牛 OS 自带的下载工具在实际使用中体验不稳定，存在“能打开但难以完成有效下载”的问题。当前需求不是做一个外部 Web 服务，而是做一个**直接运行在飞牛 OS 上的图形界面应用**，以飞牛安装包（FPK）的形式交付。

现阶段已确定的方向：

- 核心下载引擎：**Aria2 Next**
- 应用形态：**飞牛 OS GUI 应用**
- 技术路线：**Rust + Tauri + Vue 3 + TypeScript + Vite + Pinia + SQLite**
- 后期规划：**Axum** 仅在需要开放 HTTP API / 远程访问时加入

## 2. 项目背景

Motrix Next 是一个成熟的下载管理器思路模板，但飞牛 OS 的实际使用场景和桌面系统不同：

- 用户希望在飞牛系统里直接打开一个下载工具
- 工具必须有 GUI，不是只提供后台服务
- 需要更适合 NAS 场景的常驻、恢复、持久化和后台运行能力
- 需要尽量减少配置成本，让用户能“装上就用”

因此，这个项目不是简单复刻 Motrix Next，而是做一个**飞牛 OS 专用的下载工具**。

## 3. 项目目的

### 3.1 短期目标
做出一个最小可用版本，能完成以下闭环：

- 添加 HTTP / HTTPS 下载任务
- 任务开始、暂停、继续、删除
- 显示下载进度、速度和状态
- 支持下载目录设置
- 能稳定运行在飞牛 OS 上

### 3.2 中期目标
让产品具备长期使用能力：

- 任务和配置持久化
- 重启后可恢复任务状态
- 错误可追踪、日志可排查
- 更适合 NAS 环境的后台运行与自启动

### 3.3 长期目标
把它做成一个适合飞牛生态的完整下载工具：

- 支持更多协议和功能
- 支持远程管理
- 支持任务分类、计划任务、通知
- 逐步成为可维护、可扩展的下载平台

## 4. 技术栈与分工

### 4.1 Rust
**职责范围：核心业务层。**

负责：

- 业务模型定义
- 任务生命周期管理
- 配置管理
- 文件路径处理
- 启动、停止、监控 Aria2 Next
- 与前端通过 Tauri 进行命令通信

不负责：

- 页面渲染
- UI 状态展示
- 复杂前端交互

### 4.2 Aria2 Next
**职责范围：下载引擎。**

负责：

- 真正执行文件下载
- 支持 HTTP / HTTPS / BT / 磁力等下载能力
- 分片、并发、断点续传、限速等底层能力
- 通过 JSON-RPC 接受任务控制

不负责：

- 页面展示
- 任务列表 UI
- 本地持久化
- 飞牛系统交互

### 4.3 Tauri
**职责范围：应用壳与系统能力接入。**

负责：

- 创建桌面 GUI 容器
- 管理窗口、菜单、托盘
- 打包为飞牛可安装应用
- 提供前端到 Rust 的通信通道
- 调用系统文件选择、打开目录等能力

不负责：

- 下载业务本身
- 数据库存储
- 前端状态管理

### 4.4 Vue 3
**职责范围：界面层。**

负责：

- 任务列表页
- 任务详情页
- 新建任务弹窗
- 设置页
- 分类筛选
- 状态展示与交互

不负责：

- 下载引擎控制逻辑
- 数据持久化
- 后端状态同步逻辑

### 4.5 TypeScript
**职责范围：前端类型系统。**

负责：

- 任务、配置、状态等类型定义
- 前后端交互参数类型约束
- 减少页面和状态相关的低级错误

不负责：

- 运行时业务逻辑本身

### 4.6 Vite
**职责范围：前端构建工具。**

负责：

- 本地开发启动
- 热更新
- 前端构建打包
- 与 Tauri 前端工程的构建集成

不负责：

- 业务逻辑
- 状态管理

### 4.7 Pinia
**职责范围：前端全局状态管理。**

负责：

- 当前任务列表状态
- 选中任务状态
- 筛选条件
- 配置缓存
- UI 联动状态

不负责：

- 持久化数据库
- 下载引擎控制

### 4.8 SQLite
**职责范围：本地持久化。**

负责：

- 下载任务记录
- 配置存储
- 历史任务信息
- 恢复所需数据
- 错误状态记录

不负责：

- 实时 UI 状态
- 任务执行本身

### 4.9 Tokio
**职责范围：Rust 异步运行时。**

负责：

- 异步任务调度
- Aria2 RPC 通信
- 轮询任务状态
- 后台异步操作

不负责：

- 业务定义
- 数据结构设计

### 4.10 Serde
**职责范围：序列化与反序列化。**

负责：

- JSON 配置读写
- Aria2 RPC 返回值解析
- Rust 与前端传参数据转换

不负责：

- 业务逻辑

### 4.11 SQLx
**职责范围：Rust 侧数据库访问。**

负责：

- SQLite 数据读写
- 类型安全 SQL 操作
- 配置与任务记录的存取

不负责：

- 数据库部署
- UI 展示

### 4.12 tracing
**职责范围：日志与排障。**

负责：

- 关键流程日志
- 启动失败排查
- Aria2 RPC 通信排障
- 下载任务异常跟踪

不负责：

- 用户可视化界面

### 4.13 Axum（后期规划）
**当前不加入第一版。**

后期在以下情况下再引入：

- 需要局域网或远程访问
- 需要开放 HTTP API
- 需要给其他应用提供控制接口

### 开发任务


### 1.7 Aria2 Next 内置引擎策略

阶段 1 开始不再依赖飞牛 OS 或开发机系统预装的 aria2。应用默认通过 **Tauri sidecar** 内置 Aria2 Next，并仅保留 `MOTRIX_FNOS_ARIA2_PATH` 作为开发调试覆盖。

首批目标平台锁定 64 位：

- macOS Apple Silicon：`aarch64-apple-darwin`，用于当前 M1 开发测试。
- 飞牛 OS x86：`x86_64-unknown-linux-gnu`。
- 飞牛 OS ARM：`aarch64-unknown-linux-gnu`。

暂不支持 32 位 Linux：当前调研未发现飞牛 OS 必须支持 32 位内核或 32 位用户态的官方要求，且 Aria2 Next 官方预构建资产以 Linux x86_64 / Linux ARM64 为主。后续如飞牛 FPK 交付要求明确存在 32 位目标，再单独补充交叉编译和打包策略。

Aria2 Next 集成规则：

- sidecar 文件放置在 `src-tauri/binaries/`。
- Tauri 配置通过 `bundle.externalBin` 声明 `binaries/aria2-next`。
- sidecar 文件按 Rust target triple 命名，例如 `aria2-next-aarch64-apple-darwin`。
- 下载脚本必须校验 `checksums.sha256` 并设置 Unix 可执行权限。
- 启动优先级：`MOTRIX_FNOS_ARIA2_PATH` > 内置 sidecar。

## 7. 总体判断

这个项目适合按“先最小可用，再逐步增强”的路线推进。第一版不应该追求功能齐全，而应该优先把以下核心闭环做好：

- 界面可用
- 引擎可用
- 任务可用
- 持久化可用
- 在飞牛上可长期运行

## 8. 后期规划

后期如果产品形态成熟，可以继续向以下方向发展：

- 引入 Axum，开放局域网 API
- 做远程管理页面
- 提供移动端轻量控制页
- 做任务自动化调度
- 增强分类、检索和批处理能力
- 形成飞牛生态内更完整的下载管理方案

## 9. 结论

飞牛版 Motrix Next 的正确路线不是“把桌面 Motrix Next 直接搬过来”，而是：

- 用 **Rust** 建核心
- 用 **Aria2 Next** 做下载引擎
- 用 **Tauri + Vue 3** 做飞牛 GUI
- 用 **SQLite** 做本地持久化
- 以阶段性迭代方式逐步增强

这样最符合飞牛 OS 的使用场景，也最适合长期维护。
