# 飞牛版 Motrix 阶段性开发计划

> 本文档只负责记录阶段目标、具体任务、完成状态、优先级和验收标准。整体技术架构、前后端职责边界、UI 组件策略和目录规范见 `docs/architecture.md`。

> 当前进入 **FPK-first 架构纠偏期**：现有 Tauri 实现仅作为 legacy 资产参考，不再作为最终交付主线。

## 1. 项目目标

在飞牛 OS 上实现一个可安装、可运行、可维护的 GUI 下载工具，核心下载引擎采用 **Aria2 Next**，应用形态参考 Motrix Next，但面向飞牛 NAS 场景重新设计。

当前确定方向：

- 核心下载引擎：**Aria2 Next sidecar**
- 应用形态：**飞牛 OS GUI 应用**
- 基础技术路线：**Rust + Tauri + Vue 3 + TypeScript + Vite**
- 前端组件与状态：**Naive UI + Pinia**
- 本地持久化：**SQLite**
- 后期远程能力：**Axum** 仅在需要开放 HTTP API / 局域网访问时加入

## 2. 产品阶段目标

### 2.1 短期目标

做出最小可用版本，完成基础下载闭环：

- 添加 HTTP / HTTPS 下载任务
- 显示任务状态、进度、速度、大小和剩余时间
- 支持默认下载目录和单任务自定义目录
- 能在飞牛 OS 上稳定打开 GUI 并运行下载引擎
- 下载失败时能看到明确错误原因

### 2.2 中期目标

让应用具备长期使用能力：

- 任务暂停、继续、删除
- 任务和配置持久化
- 应用重启后恢复任务状态
- 应用内调试日志与问题排查能力
- 更适合 NAS 环境的后台运行、自启动和通知

### 2.3 长期目标

逐步发展为完整下载管理工具：

- 支持 BT、磁力、批量添加任务
- 支持分类、筛选、计划任务和自动化
- 支持局域网或远程管理
- 支持更完整的任务历史和错误追踪
- 最终以飞牛生态可安装应用形态交付

- 应用重启后会从 SQLite 加载任务，并在 Aria2 RPC ready 后刷新恢复状态。

- 明确退出时会停止后台任务同步，退出前同步一次 Aria2 状态，暂停未完成任务并保存到 SQLite，再停止当前 AppState 管理的 Aria2 sidecar。
- AppState 已记录 Aria2 运行态端口、RPC secret、endpoint、PID 和退出状态，后端 RPC 调用使用运行时实际配置。
- Aria2 sidecar 启动已支持首选 6800 和备用 16800-16820；外部占用会跳过，确认属于本应用的残留 sidecar 才尝试清理复用。
- 已提供设置弹窗，支持开机自启、下载通知、最大并发下载数、下载限速和上传限速。
- 下载完成和失败会在通知开关开启后触发系统通知，后端后台监控负责状态同步与通知去重。
- SQLite 中保存的并发与限速配置会在启动 RPC ready 后应用到 Aria2，设置保存后也会即时尝试应用。

当前问题：

- 暂未提供独立设置页和历史记录页；配置与历史数据已落库，后续阶段再补 UI 入口。
- 重启恢复依赖已有 Aria2 GID；若 Aria2 侧已丢失任务，当前会标记为错误并记录日志。
- 外部强杀、断电、系统崩溃等场景仍无法保证执行退出清理，只能依赖启动时残留 sidecar 识别与端口兜底。

当前处理原则：

- 前端新增任务能力继续进入 `src/features/tasks/`。
- 诊断/日志能力继续进入 `src/features/diagnostics/`。
- 后续实现必须遵守 `docs/architecture.md`，不得重新向 `MainWindow.vue` 堆叠复杂 UI 或业务逻辑。

## 5. 阶段 1：最小可用下载器（✅ 已完成，任务控制后移阶段 4）

### 5.1 目标

实现一个能真正下载 HTTP / HTTPS 文件的最小版本。

### 5.2 已完成任务

#### 5.2.1 Aria2 Next 内置引擎策略

- 默认通过 Tauri sidecar 内置 Aria2 Next。
- 保留 `MOTRIX_FNOS_ARIA2_PATH` 作为开发调试覆盖。
- sidecar 文件放置在 `src-tauri/binaries/`。
- Tauri 配置通过 `bundle.externalBin` 声明 `binaries/aria2-next`。
- sidecar 文件按 Rust target triple 命名，例如 `aria2-next-aarch64-apple-darwin`。
- 下载脚本校验 `checksums.sha256` 并设置 Unix 可执行权限。
- 启动优先级：`MOTRIX_FNOS_ARIA2_PATH` > 内置 sidecar。

首批目标平台：

- macOS Apple Silicon：`aarch64-apple-darwin`，用于当前 M1 开发测试。
- 飞牛 OS x86：`x86_64-unknown-linux-gnu`。
- 飞牛 OS ARM：`aarch64-unknown-linux-gnu`。

暂不支持 32 位 Linux。后续如飞牛 FPK 交付要求明确存在 32 位目标，再补充交叉编译和打包策略。

#### 5.2.2 新建下载任务

- 输入下载链接。
- 支持单个 HTTP / HTTPS 链接。
- 支持设置文件名和保存路径。
- 创建 Aria2 Next 下载任务。

#### 5.2.3 任务列表与状态同步

- 显示全部任务。

- 同步 Aria2 GID、保存目录、文件路径、错误码和错误原因。
- 任务区域支持横向滚动，列表区域内部纵向滚动。

#### 5.2.4 下载过程展示

- 显示下载百分比。
- 显示实时速度。
- 显示剩余时间。
- 显示已下载大小和总大小。

#### 5.2.5 下载目录设置

- 默认目录不存在时自动创建。
- 支持单任务自定义目录。

- 检查目录是否可创建和可用。

#### 5.2.6 错误反馈

## 7. 阶段 3：日志调试系统

### 7.1 目标

在不能使用开发终端的飞牛 / FPK / 生产环境中，提供可视化日志窗口，帮助定位 Aria2 sidecar、RPC、证书、路径权限和下载失败等问题。

### 7.2 开发任务

#### 7.2.1 内存日志队列

- 在 Rust `AppState` 中增加内存日志环形队列。
- 日志至少包含时间、级别、来源模块、消息。
- 默认保留最近 500 条，避免长期占用内存。

#### 7.2.2 日志写入点

- 记录应用启动和后端通信状态。
- 记录 Aria2 sidecar 启动来源、启动参数摘要和进程状态。
- 记录 CA 证书探测结果。
- 记录默认下载目录解析和目录创建结果。
- 记录 `aria2.addUri` 创建结果。
- 记录 `aria2.tellStatus` 返回的错误码和错误原因。
- 记录 RPC ready / timeout / 连接失败。

#### 7.2.3 Tauri 日志命令

- `list_debug_logs`：读取当前内存日志。
- `clear_debug_logs`：清空当前内存日志。
- 后续预留 `export_debug_logs`：导出日志文本。

#### 7.2.4 前端调试日志窗口

- 在诊断入口中增加“调试日志”窗口。
- 支持按时间顺序展示 info / warn / error。
- 支持复制全部日志。
- 支持清空日志。
- 支持自动滚动到底部。

#### 7.2.5 后续设置预留

- 后期在设置页中增加是否展示调试入口。
- 后期增加是否记录到文件。
- 后期增加日志保留天数和导出路径。
- 落盘日志必须避免记录敏感信息，如 RPC secret 和完整私密下载链接 query。

### 7.3 完成标准

- 非 dev 环境中可以打开调试日志窗口。
- 下载失败时能从日志中看到关键错误链路。
- Aria2 启动、RPC、目录、证书问题可通过日志定位。
- 日志窗口支持复制和清空。

状态：✅ 已完成。

## 12. 验收原则

每个阶段完成后必须满足：

- 功能可手动验证。
- 不破坏已有下载闭环。
- 文档状态同步更新。
- 新代码符合 `docs/architecture.md`。
- 如果涉及前端 UI，不能继续向 `MainWindow.vue` 堆叠复杂实现。
- 如果涉及生产环境问题，必须考虑日志和用户可见错误。

## 13. 总体判断

项目应继续按“先最小可用，再治理架构，再逐步增强”的路线推进。

当前最关键的问题不是继续加新功能，而是先把前端架构和任务表实现方式拉回可维护状态。任务表列宽拖动、组件库使用、Pinia 状态管理和 `MainWindow.vue` 拆分应作为下一步优先工作。
